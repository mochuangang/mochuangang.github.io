1.Hive基本概念

1.1 什么是Hive
Hive：由Facebook开源用于解决海量结构化日志的数据统计工具。
Hive是基于Hadoop的一个数据仓库工具，可以将结构化的数据文件映射为一张表，并提供类SQL查询功能。
本质是：将HQL转化成MapReduce程序

1）HDFS：Hive处理的数据存储在HDFS上[x]
2）MR： Hive作业(SQL)是通过Hive的框架翻译成MR作业
还支持Tez Spark引擎；
不管底层到底是使用什么引擎，对于用户来说是不感知
同样的SQL，只需要通过一个参数切换，就可以运行不同的执行引擎之上
3）YARN： Hive的作业是提交到YARN上面去运行的

Hadoop：开发可以使用单机，生产上肯定都是分布式的
Hive其实就是一个客户端（提交机器）而已，没有集群的概念
SQL ==> Hive ==> MR ==> YARN

Hive职责：将SQL翻译成底层对应的执行引擎作业
统一的元数据（metadata）管理 == metastore
元数据：描述数据的数据 schema

Hive
数据是存放在distributed storage上的
元数据是存储在metastore所对应的底层数据库中的(MySQL)

2.Hive架构 （SQL on Hadoop）

  1. 用户接口：Client
    CLI（command-line interface）、JDBC/ODBC(jdbc访问hive)、WEBUI（浏览器访问hive）

  2. 元数据：Metastore
    元数据包括：表名、表所属的数据库（默认是default）、表的拥有者、列/分区字段、表的类型（是否是外部表）、表的数据所在目录等；
    默认存储在自带的derby数据库中，推荐使用MySQL存储Metastore

  3. Hadoop
    使用HDFS进行存储，使用MapReduce进行计算。

  4. 驱动器：Driver
    （1）解析器（SQL Parser）：将SQL字符串转换成抽象语法树AST，这一步一般都用第三方工具库完成，比如antlr；对AST进行语法分析，比如表是否存在、字段是否存在、SQL语义是否有误。
    （2）编译器（Physical Plan）：将AST编译生成逻辑执行计划。
    （3）优化器（Query Optimizer）：对逻辑执行计划进行优化。
    （4）执行器（Execution）：把逻辑执行计划转换成可以运行的物理计划。对于Hive来说，就是MR/Spark。

Hive通过给用户提供的一系列交互接口，接收到用户的指令(SQL)，使用自己的Driver，结合元数据(MetaStore)，将这些指令翻译成MapReduce，提交到Hadoop中执行，最后，将执行返回的结果输出到用户交互接口。

Hive vs RDBMS

  • 分布式
  • 节点数
  • 成本
  • 数据量
  • insert update delete
  • 事务
  • 延时性：批处理/离线处理

Hive适用场景

  • 批处理/离线处理
  • 延时性很大的
  • 尽量少涉及到update或者delete这种操作，虽然是支持的

Hive优缺点
优：易上手、比MR使用起来简单的多
缺：高延时

6.遇到的问题

再打开一个客户端窗口启动hive，会产生java.sql.SQLException异常。

原因是，Metastore默认存储在自带的derby数据库中,
derby是一个单session,只能单独开一个窗口，生产中推荐使用MySQL存储Metastore;
